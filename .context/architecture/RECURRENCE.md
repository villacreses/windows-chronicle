# Recurrence

Chronicle's recurrence engine is a projection system with first-class
addressable derived entities. A recurring master row plus its rule
defines an addressable space of occurrences; the expander materializes
the slice intersecting the current load range; persistent decorations
(EXDATE, `EventOverride`) modify that space at known anchors. **No
occurrence is ever persisted.**

## Storage and Parsing

Recurrence is stored as RFC 5545 RRULE strings on the master event
(e.g. `FREQ=WEEKLY;BYDAY=MO,WE;UNTIL=20261231T000000Z`).

The same format is spoken natively by Google, Outlook (via Graph),
Apple Calendar, and CalDAV. Future provider adapters become a
field-mapping exercise rather than a translation one.

Supported RRULE surface in MVP:

- `FREQ` — Daily, Weekly, Monthly, Yearly.
- `INTERVAL`.
- `BYDAY` (weekly).
- `BYMONTHDAY`.
- `COUNT`.
- `UNTIL`.

Parsing produces a `RecurrenceRule` value object under
`Models/Recurrence/`.

Deferred from MVP: ordinal BYDAY, multi-value BYMONTHDAY, RDATE,
custom RRULE input.

## EXDATE — The Hole

EXDATE is the persistent representation of "skip this occurrence." It
is a list of UTC anchors on the master event. The expander filters
anchors against this list during the per-anchor pipeline.

EXDATE is a *semantic constraint over the projection space*, not a
property of the algorithm: each entry must equal a walker-emitted
anchor bit-for-bit. The write path persists `SeriesAnchorUtc`
verbatim — no normalization, no rounding.

The "Skip this occurrence" UI gesture appends `SeriesAnchorUtc` to the
master's EXDATE list and writes the master back.

## EventOverride — The Divergence

`EventOverride` is the persistent representation of "this one
occurrence is different." It is a row in `EventOverrides` keyed by
`(SeriesEventId, OccurrenceAnchorUtc)`, with per-field nullable
columns. A null column means "inherit from master at expansion time."

There is no `IsCancelled` column. Cancellation is EXDATE's job;
overrides handle modification only. This split exists because EXDATE
and overrides have different semantics: EXDATE creates a hole; an
override creates a divergence that must be tracked and merged
forever.

The expander's merge step joins overrides into the per-anchor pipeline
after the EXDATE filter and before the merged-range gate. Walk
termination is extended by `maxPastDisplacement` so overrides that
pull future anchors back into the visible window are reached.

## Named Invariants

These are load-bearing. Any change that violates them is a breaking
change to the projection space and requires data migration.

1. **Occurrence identity = `(Event.Id, SeriesAnchorUtc)`.** Expanded
   occurrences carry their master's `Id`; `SeriesAnchorUtc` is the
   per-occurrence discriminator. `Event.Id` is unique among rows in
   the `Events` table but NOT among in-memory `Event` instances. Code
   that needs a key over a mixed collection must use `EventKey.For(evt)`.
   `SeriesAnchorUtc` and `StartTimeUtc` may differ on the same
   occurrence — an override can move the displayed wall-clock Start
   while identity stays on the rule-walk anchor. Any write keyed on
   "this occurrence" (EXDATE append, override upsert, scope-picker
   dispatch) must use `SeriesAnchorUtc`, not `StartTimeUtc`.
   `EventRef.From(evt)` constructs the correct key.

2. **Identity stability is rule-version conditional.** `EventKey` is
   stable across loads, view switches, and cache lifecycle — but only
   within a single recurrence rule version. Semantic changes to RRULE
   handling, the anchor-walk algorithm, or the time-zone evaluation
   strategy are breaking; pre-existing EXDATEs and overrides become
   undefined.

3. **EXDATE is a semantic constraint over the space, not the
   algorithm.** Each EXDATE entry must equal a walker-emitted anchor
   bit-for-bit. The write path persists `SeriesAnchorUtc` verbatim.
   Today the space and algorithm coincide on UTC equality; the
   tz-aware walk preserves that coincidence intentionally (the walker
   emits UTC anchors in both modes).

4. **Cache fields are advisory.** `RecurrenceEndUtcCached` is used
   solely by `EventRepository.GetInRangeAsync` to prune ended finite
   series. The expander never reads it; it is not authoritative
   metadata and never participates in expansion or termination
   decisions. If a cached value disagrees with the rule, the rule wins
   and the cache is a bug to be repaired by the writer.

5. **`_eventsByDate` is a render-time projection cache, never an
   identity source.** UI state that needs to remember a selection
   across reloads must hold an `EventKey`, not a dictionary slot or
   list index.

6. **COUNT counts pre-EXDATE.** Per RFC 5545, COUNT counts all
   instances generated by the rule, before EXDATE filtering. Filter
   EXDATE after the count walk terminates.

7. **The expander never throws on bad timezone data.** A `TimeZoneId`
   that fails to resolve (`TimeZoneNotFoundException` /
   `InvalidTimeZoneException`) degrades to the legacy UTC walk for
   that one series, with a `Debug.WriteLine`. One malformed row cannot
   take down the load. The write boundary (`Event.Validate` plus
   `EventEditPopover.GetDefaultRecurringTimeZoneId`) prevents
   Chronicle itself from writing bad data; the expander fallback
   handles bad data from other paths (manual edits, future imports,
   restored backups). Both ends of the defense matter.

8. **Single walk dispatch.** `RecurrenceExpander.Expand` and
   `RecurrenceExpander.ComputeEndUtc` both route through one
   `WalkAnchorsForMaster(start, rule, timeZoneId)` helper. The tz-aware
   branch (and any future walk-strategy variation) lives inside that
   one helper. Drift between the rendering walk and the cached-end
   walk is structurally impossible — there is no parallel
   implementation to fall out of sync.

## Expansion Logic

`RecurrenceExpander.Expand` runs after the repository returns
recurring masters and before `_eventsByDate` is built. The pipeline
per master:

1. `WalkAnchorsForMaster` enumerates rule anchors (UTC or
   tz-projected — see DST below).
2. Each anchor is checked against the rule's termination conditions
   (`COUNT`, `UNTIL`, walk-range end).
3. Anchors in the master's EXDATE list are filtered out.
4. Overrides keyed at this anchor are merged in — per-field nullable
   columns overlay the master's defaults.
5. Anchors outside the merged event's range (after override
   displacement) are filtered out.
6. Surviving anchors are emitted as transient `Event` instances
   carrying the master's `Id`, the anchor as `SeriesAnchorUtc`, and
   merged field values.

Walk termination is extended by `maxPastDisplacement` so overrides
that pull future anchors back into the visible window are reached.

`RecurrenceExpander.ComputeEndUtc` reuses `WalkAnchorsForMaster` to
compute a finite series's terminal UTC for `RecurrenceEndUtcCached`.
The two consumers share one walk implementation by invariant #8.

## Time Zone Anchoring and DST

Recurring events optionally carry an IANA `TimeZoneId` as their anchor
frame:

- `TimeZoneId IS NULL` — legacy UTC anchoring; the walker steps in
  UTC. A weekly meeting created at 9 AM local will drift ±1 hour
  across DST.
- `TimeZoneId` set — wall-clock anchoring; the walker steps in local
  time and projects each anchor to UTC at emission. "Weekly Monday 9
  AM" stays at 9 AM local across DST.

`WalkAnchorsForMaster` is the single dispatch (invariant #8). The
tz-aware branch caches one `TimeZoneInfo` per `Expand` call — the
documented exception to the engine-level "no `TimeZoneInfo` lookups in
per-event loops" rule. EXDATE writes and override identity remain
keyed on the UTC anchors the walker emits in either mode; the
read-side contract is mode-agnostic.

### DST Edge Cases

Matching Google / Apple / libical convention:

- **Invalid local times (spring-forward gap).** `ResolveLocalForDst`
  shifts forward by the DST adjustment delta so the event still
  occurs on its scheduled day. Applied before
  `TimeZoneInfo.ConvertTimeToUtc`.
- **Ambiguous local times (fall-back overlap).**
  `TimeZoneInfo.ConvertTimeToUtc` defaults to standard time (the
  earlier interpretation). We don't override.

### `TzWalkPad`

A 1-hour pad added to walk-termination gates (`walkEndUtc`, `UNTIL`)
when a tz-aware walk is active.

The spring-forward shift produces a one-time UTC blip at DST
transitions that would otherwise cause premature termination. Anchors
temporarily past unpadded UNTIL are skipped (continue) without
counting toward COUNT or updating `lastAnchor`, preserving rule
semantics.

## Anchor Zone Is Authoritative

A tz-aware master records the zone it was created in. That zone is
the series's anchor for its entire lifetime.

- Display always converts to `TimeZoneInfo.Local`.
- Edits never re-anchor — saving from a different OS zone preserves
  the master's existing `TimeZoneId`.

A "9 AM NY meeting" edited from a PT machine remains anchored to NY
and renders as 6 AM PT to the editor.

This is a deliberate product position, not an implementation
artifact:

- "Re-anchor to current system zone on edit" would silently invalidate
  every EXDATE and override on the series (invariant #2) every time
  the user travelled and touched the event.
- "Treat display zone as authoritative" removes the concept of a
  fixed anchor entirely; the same calendar would render differently
  in two locations and the EXDATE / override identity contract would
  have nothing stable to key on.
- The traveler case the chosen position handles imperfectly ("I moved
  to LA permanently and now my NY meetings show at the wrong local
  time") is recoverable by recreating the series — explicit user
  intent for an explicit semantic change. The traveler cases it
  handles correctly ("I'm visiting LA for a week and my recurring NY
  meetings should still happen at 9 AM NY") cover the common need.

A maintainer reading the edit-path code and seeing
`TimeZoneId = eventToEdit.TimeZoneId` preserved across saves is
looking at this decision, not an oversight.

There is no auto-migration between UTC and tz-aware anchoring on an
existing master. Such a migration would change the projection space's
equivalence relation (invariant #2). Users who want the wall-clock
fix recreate the series.

New recurring events default `TimeZoneId` to the system's current
zone via `EventEditPopover.GetDefaultRecurringTimeZoneId`, which
normalizes Windows IDs → IANA at the write boundary. `Event.Validate`
rejects strings that don't resolve via `TimeZoneInfo` as
defense-in-depth.

## Recurrence-Aware Mutation

The user-facing scope picker presents three options (This event / All
events / Cancel). Whichever the user chooses, the write reduces to
one of these shapes:

- **Edit one occurrence.** `EventEditPopover.ShowEditOccurrenceAsync`
  → `OverrideRepository.UpsertAsync(EventRef.Occurrence, OverrideFields)`.
  Per-field nullable columns; null = inherit.
- **Edit the whole series.** `EventEditPopover.ShowEditEventAsync` on
  the master → `EventRepository.UpdateAsync`. Preserves the master's
  existing `TimeZoneId` per the authoritative-anchor position above.
- **Skip one occurrence.** Append `SeriesAnchorUtc` to the master's
  EXDATE list verbatim and write the master.
- **Delete the whole series.**
  `EventRepository.DeleteAsync(occurrence.Id)` works because
  `occurrence.Id == master.Id`; the override cascade helpers clean up
  `EventOverrides` rows.

The persistence boundary is enforced by
`EventRepository.RefuseOccurrence` at the repository chokepoint;
`OverrideRepository.UpsertAsync` accepts only `EventRef.Occurrence`,
making the master variant a compile error.

## Tolerated Ambiguities

Three corners deliberately not enforced; the write-path UX prevents
normal user paths from producing them.

### Dual `Event` Semantics in UI Read Paths

`Event` carries two semantically distinct shapes — persistent rows and
expanded occurrences — in a single type. The distinction is mediated
by `Event.IsOccurrence` and `EventKey` at read sites and enforced at
the persistence boundary by `EventRepository.RefuseOccurrence`.

A wrapper type (`EventInstance` or similar) that encodes the
projection-vs-entity split in the type system was discussed and
deliberately deferred. The cost of a wrapper is paid in every renderer
and tap-target signature; the bug class it would structurally prevent
(occurrence handed to a persistence write) is already caught by the
repository guard. `EventRef` is the narrower form chosen for mutation
boundaries specifically — where the discrimination is actionable.

Revisit trigger: a second projection consumer needing the distinction
at read time (drag/drop reschedule, inline edit, multi-select
copy/paste, provider sync reconciliation), or `IsOccurrence` branching
appearing in more than one mutation path beyond the existing edit /
delete / override-save handlers.

### Merge Validity Is Not Enforced Post-Merge

`EventOverride.Validate` catches `End < Start` only when both override
times are set. The expander does not re-validate the merged `Event`
before emitting. A malformed override smuggled in via direct SQL (or
a future writer that bypasses the editor's validation) can produce an
emitted occurrence with `End < Start`, or `IsAllDay = true` paired
with non-zero Start/End times. Renderer behavior on such malformed
merges is undefined. The editor form's own validation prevents normal
user paths from producing these. Acceptable for MVP; revisit if a
non-UI writer is introduced.

### Master-Loading Gap for Cross-Boundary Overrides

Override completeness within the loaded master set is explicit. But
masters themselves may be filtered out by
`EventRepository.GetInRangeAsync` based on the master's natural
boundaries (`StartTimeUtc > rangeEnd` or
`RecurrenceEndUtcCached < rangeStart`). If an override on such a
filtered-out master would pull an occurrence into the visible window,
that occurrence is silently dropped — the master isn't loaded, so its
overrides aren't queried, so the displacement isn't computed.

Both ends of the gap require the user to have moved an occurrence
across the master's *natural* boundary, which is unusual. The fix is
a two-pass load (query candidate overrides first, then load masters
including those referenced) and is deferred until a user actually
hits the limitation.
